# 실험 1 — Feature 명세서

`exp1_dataset.csv` (3,696행 × 17컬럼)의 각 컬럼이 무엇이고, 어디서 왔고, 어떻게 계산되는지.

---

## 0. 전제 — 모든 feature는 "쌍" 단위다

재료 하나의 속성이 아니라 **(herb A, candidate B) 조합**에 대한 값이다.

```
herb A        한약재 14개 (강황·당귀·대추·마·모과·박하·생강·시나몬
                          ·연꽃·우엉·월계수·칡·팔각·회향)
candidate B   후보 264개 (5소스 top-20의 합집합, 중복 제거)

14 × 264 = 3,696행
```

원본 데이터는 재료별 속성인 경우가 많으므로(예: 성미귀경), **쌍 관계로 변환하는 과정**이 들어간다.

```
대추   성 +1(溫)  미 {甘}   귀경 {胃,脾,心}
생강   성 +1(溫)  미 {辛}   귀경 {肺,脾,胃}
                    ↓ 변환
qi_diff 0 · qi_B 1 · wei_jac 0.0 · gui_jac 0.5
```

## 1. 조리 축 (3-dim) — RecipeDB 430 레시피

### 원본

`recipedb_counts_only.json`

```json
{ "herb_ko": "대추",
  "recipe_count": 30,
  "ingredient_counts": {"sugar": 18, "butter": 14, "ginger": 6, ...} }
```

→ 대추가 등장한 레시피 30개 중, 생강이 6번 함께 등장.

### `cook_count`

```
cook_count = log1p(동시등장 레시피 수)
           = log1p(6) = 1.79
```

로그를 씌우는 이유: 설탕(18)과 생강(6)의 raw 차이가 크면 큰 값이 학습을 지배한다.

**범위** 0 ~ 약 3.0

### `cook_rank_inv`

```
cook_rank_inv = 1 / rank
```

생강이 대추의 co-occurrence 4위 → `1/4 = 0.25`. top-N 밖은 0.

**범위** 0 ~ 1.0 (1위 = 1.0, 20위 = 0.05)

> **주의**: herb 내 rank가 count 순으로 매겨지므로 `cook_count`와 **거의 중복**이다.
> 상관계수가 매우 높을 것으로 예상되며, 조리 축의 실질 정보는
> `빈도` + `PMI` 두 종류로 봐야 한다.

### `cook_pmi`

```
PMI = log( P(A,B) / (P(A) · P(B)) )
PPMI = max(0, PMI)
```

분모는 **RecipeDB 내부에서 추정**한다 (RecipeNLG를 쓰지 않는 것이 실험 1의 제약).

```
N = 430
P(herb)   = recipe_count / 430
P(B)      = B가 등장한 총 레시피 수 / 430
P(herb,B) = 동시등장 count / 430
```

**안정화 처리** (표본 430개이므로 필수)

- add-0.5 smoothing
- `c_ab < 2`인 쌍은 `pmi = 0`으로 처리 (1회 동시등장은 우연)
- 음수 클리핑 (PPMI)

**역할**: 전역 빈도를 걷어낸다.

설탕은 대추와 18번 등장(count 1위)하지만 430개 레시피 대부분에 나온다.
따라서 count는 최고지만 **PMI는 낮아야 정상**이다. PMI가 제 역할을 하는지
확인하려면 설탕의 PMI가 실제로 낮게 나오는지 보면 된다.

### 조리 축 커버리지

```
mask_cook = 1 :  1,414 / 3,696  (38.3%)
positive 쌍 중:    281 / 458    (61.4%)
```

## 2. 향미 축 (2-dim)

### `fg_cosine` — FlavorGraph

Park et al. (2021, *Scientific Reports*)의 사전학습 임베딩.
8,297 노드(ingredient 6,653 + compound 1,644), 300차원. **ingredient 노드만 사용.**

```
fg_cosine = cos(emb_A, emb_B)
```

**정합성 검증 완료**

| 쌍 | 코사인 | 판정 |
|---|---:|---|
| basil ~ oregano | 0.364 | 높음 (정상) |
| ginger ~ chocolate | 0.014 | 낮음 (정상) |
| 실측 PMI와의 상관 | ρ=0.215 | 양수 (로드 정상) |

**연속값**이므로 세 축의 feature 중 정보량이 가장 풍부하다.

**노드명 보정 이력**

| herb | 초기 매칭 | 수정 | 근거 |
|---|---|---|---|
| 연꽃 | lotus root (실패) | `lotus_seed` | 우리 데이터가 연자육 기준 |
| 우엉 | burdock (실패) | `burdock_root` | 식용 뿌리 기준 |

보정 후 14개 herb 전부 매칭 성공.

### `fdb_rank_inv` — FlavorDB

```
fdb_rank_inv = 1 / rank   (FlavorDB top-20 내), 없으면 0
```

**알려진 결함 — 곡물 폴백**

화학 데이터가 부족한 재료에서 동일한 fallback이 나온다.

```
인삼 / 국화 / 대추 / 마  →  top-5가 전부 "빵·보리·귀리·쌀·옥수수"
```

따라서 이 feature는 커버리지가 낮고 신뢰도가 균일하지 않다.
(대추의 top-20에 생강이 없어 `fdb_rank_inv = 0`인 것도 이 때문)

### 향미 축 커버리지

```
mask_flavor = 1 :  1,904 / 3,696  (51.5%)
positive 쌍 중:      372 / 458    (81.2%)
```

**positive 쌍에서 세 축 중 커버리지가 가장 높다.**

## 3. 전통 축 (4-dim) — TCM-MKG 성미귀경

### 원본

Zeng & Jia (2025), *J Pharm Anal* 101342. Zenodo 13763953. CC BY-NC 4.0.

| 파일 | 내용 |
|---|---|
| `herb_properties_tcmmkg.csv` | herb A 14개 (월계수 제외 13개 보유) |
| `candidate_b_properties_v2.csv` | candidate B, 기각 6건 반영 |

**세 가지 속성**

| 속성 | 형식 | 값 |
|---|---|---|
| 성(性) | 수치 1개 | 寒 −2 / 涼 −1 / 平 0 / 溫 +1 / 熱 +2 |
| 미(味) | 집합 | 酸 苦 甘 辛 鹹 澀 (6종, 복수 가능) |
| 귀경(歸經) | 집합 | 肺 肝 心 脾 腎 胃 膀胱 膽 大腸 小腸 心包 三焦 (12경) |

원본 D7이 long format(한 약재가 여러 행)이며 영문 표기(`Sweet medicinal`,
`Liver meridian`)로 되어 있어 파싱 후 집합화한다.
성(性)은 이미 수치화되어 있어 별도 매핑 불필요.

### `qi_diff` — 온열 차이

```
qi_diff = |qi_A − qi_B|
```

| 쌍 | 계산 | 값 |
|---|---|---:|
| 생강(溫+1) + 대추(溫+1) | \|1−1\| | 0 |
| 시나몬(熱+2) + 박하(涼−1) | \|2−(−1)\| | 3 |
| 당귀(溫+1) + 꿀(平 0) | \|1−0\| | 1 |

**의미**: 0이면 같은 성질끼리 강화(溫+溫 = 온보), 크면 서로 견제(熱+涼 = 중화).
방제 데이터에서 생강↔대추가 75회로 최상위인데, 이것이 계지탕 계열의 溫+溫 조합이다.

**범위** 0 ~ 4

### `qi_B` — 후보의 온열

```
qi_B = candidate의 성 수치 그대로
```

**왜 따로 넣는가**: `qi_diff`만으로는 방향을 알 수 없다.

```
당귀(+1) + 후보(0)   → qi_diff = 1
당귀(+1) + 후보(+2)  → qi_diff = 1   ← 같은 값인데 완전히 다른 조합
```

**`qi_A`를 넣지 않는 이유**: Leave-One-Herb-Out에서 test fold는 herb 하나뿐이므로
`qi_A`가 상수가 된다. 해당 fold 안에서 아무 정보도 주지 못한다.
A의 정보는 `qi_diff`에 간접적으로 들어간다.

**범위** −2 ~ +2

### `wei_jac` — 미(味) 겹침

```
wei_jac = |味_A ∩ 味_B| / |味_A ∪ 味_B|
```

| 쌍 | 계산 | 값 |
|---|---|---:|
| 당귀{甘,辛} + 시나몬{甘,辛} | 2/2 | 1.0 |
| 당귀{甘,辛} + 대추{甘} | 1/2 | 0.5 |
| 당귀{甘,辛} + 모과{酸} | 0/3 | 0.0 |

**의미**: 味는 맛이자 약리 작용이다.

| 味 | 작용 | 예 |
|---|---|---|
| 甘 | 보하고 완화 | 대추, 마, 감초 |
| 辛 | 발산하고 퍼뜨림 | 생강, 박하 |
| 酸 | 수렴하고 모음 | 모과, 오미자 |
| 苦 | 내리고 말림 | 강황 |
| 澀 | 붙잡아 둠 | 연자육 |

14개 herb 중 **모과만 酸 단독**이라 나머지 12개와 味가 하나도 겹치지 않는다.
모델에서 특이점으로 작동할 수 있다.

**범위** 0 ~ 1

### `gui_jac` — 귀경(歸經) 겹침

```
gui_jac = |歸經_A ∩ 歸經_B| / |歸經_A ∪ 歸經_B|
```

| 쌍 | 교집합 | 합집합 | 값 |
|---|---|---:|---:|
| 당귀{脾,肝,心} + 대추{胃,脾,心} | {脾,心} | 4 | 0.50 |
| 당귀{脾,肝,心} + 칡{胃,脾} | {脾} | 4 | 0.25 |
| 박하{肺,肝} + 대추{胃,脾,心} | ∅ | 5 | 0.00 |

**의미**: 몸의 어느 계통에 작용하는가. 겹치면 같은 계통에 함께 작용해 효과가
강해지거나 과해지고, 안 겹치면 서로 다른 영역을 담당해 보완이 된다.
실제 방제의 군신좌사(君臣佐使) 구성이 이 논리로 짜인다.

**관문 B에서 세 속성 중 유일하게 유의했던 축** (ρ=0.135, p=5.4×10⁻⁷).
다만 단조 관계가 아니라 예측 변수로는 부적합하다.

| gui_jac 구간 | 쌍 수 | 평균 레시피 유사도 |
|---|---:|---:|
| 0.00~0.01 | 345 | 0.128 |
| 0.20~0.40 | 558 | 0.150 |
| **0.40~0.60** | 249 | **0.191** |
| 0.60~1.00 | 160 | 0.158 |

가장 비슷한 최상위 구간이 중간 구간보다 낮다.

**범위** 0 ~ 1

### 전통 축 커버리지

```
mask_prop = 1 :  1,053 / 3,696  (28.5%)
positive 쌍 중:    244 / 458    (53.3%)
```

`1,053 = 81 × 13` — 성미 완비 후보 81개 × 월계수 제외 13개 herb.

> **미해결 항목**: 이전 B축 매핑에서는 성미 보유가 116개(기각 6건 제외 후 110개)였다.
> "성·미·귀경을 모두 갖춘" 완비 기준을 적용하면서 81개로 줄었다.
> 4개 feature는 부분적으로도 계산 가능하므로(歸經이 없어도 성·미로 3개는 계산 가능),
> feature별 마스크로 쪼개면 커버리지가 올라갈 수 있다.

## 4. 마스크 (3-dim) — 0과 결측의 구분

**값이 아니라 "이 축이 발언 가능한가"를 나타낸다.**

```
mask_cook   = candidate가 RecipeDB 어휘(825종)에 존재
mask_flavor = herb, candidate 둘 다 FlavorGraph 노드에 존재
mask_prop   = herb, candidate 둘 다 성미귀경 보유
```

### 왜 필요한가

마스크가 없으면 `0`이 두 가지를 뜻하게 된다.

```
mask 없음:  전통 feature = [0, 0, 0, 0]
            → "성질이 안 맞다"인가 "정보가 없다"인가 구분 불가
            → 게이트가 "전통 0.15"를 내도 해석이 안 됨
```

### 핵심 원칙 — count 0은 마스크가 아니다

```
버터가 RecipeDB 어휘에 있고, 대추와 함께 등장한 적이 없음
  → cook_count = 0, mask_cook = 1
  → "함께 안 쓰인다"는 유효한 정보

황기가 RecipeDB 어휘에 아예 없음
  → mask_cook = 0
  → 판단할 근거 자체가 없음
```

### 모델에서의 처리

```python
logits = gate(concat(features, mask))
logits = logits.masked_fill(~mask, -1e9)
w = softmax(logits)          # 마스크된 축은 정확히 0, 나머지로 재정규화
```

```
mask = [1, 1, 0]
logits = [2.1, 1.3, -1e9]
w = [0.69, 0.31, 0.00]
```

부작용: **한 축만 활성이면 게이트가 자동으로 1.0이 된다.** 선택한 것이 아니라
선택지가 없는 것이므로 해석 시 주의.

### 대조 사례

```
(대추, 버터)
  조리   count 2.3 · rank_inv 0.5 · pmi 0.1      mask 1
  향미   fg 0.21 · fdb 0                          mask 1
  전통   —                                         mask 0   버터는 본초 약재 아님

(대추, 황기)
  조리   —                                         mask 0   서양 레시피에 없음
  향미   —                                         mask 0   FlavorGraph에 없음
  전통   qi_diff 0 · qi_B 1 · wei 0.5 · gui 0.33   mask 1

(대추, 안초비)
  세 축 모두 mask 0                               ← 어떤 근거로도 판단 불가
```

### 축 조합 교차표

| cook | flavor | prop | 전체 | positive |
|:---:|:---:|:---:|---:|---:|
| 1 | 1 | 1 | 481 | 167 |
| 1 | 1 | 0 | 891 | 113 |
| 1 | 0 | 1 | 0 | 0 |
| 1 | 0 | 0 | 42 | 1 |
| 0 | 1 | 1 | 273 | 54 |
| 0 | 1 | 0 | 259 | 38 |
| 0 | 0 | 1 | 299 | 23 |
| **0** | **0** | **0** | **1,451** | **62** |

**주목할 점**

- `cook=1, flavor=0, prop=1`이 **0쌍** — RecipeDB에 있는 재료는 거의 다
  FlavorGraph에도 있다. 두 축의 관할이 97% 겹친다(1,372 / 1,414).
  실질적으로 갈리는 것은 전통 축뿐이다.
- **세 축 모두 부재 1,451쌍(39.3%)** — 학습에서 제외되며, 이 개수 자체가
  "어떤 근거로도 접근 불가한 페어링"을 나타내는 지표다.

관련 후보 125종의 성격:

| 분류 | 예시 |
|---|---|
| 서양·비아시아 식재 | 블랙커런트, 프로슈토, 안초비, 러비지, 케이퍼 |
| 가공·복합조미 | 오향분, 가람마살라, 해선장, 카레가루, 젤라틴 |
| 육수·유지 | 닭육수, 채소육수, 카놀라유, 코코넛오일 |
| 치즈류 | 체더, 블루, 스위스, 모차렐라, 부라타 |
| herb 측 부재 | 월계수 전체(14쌍) — 본초 미수재 |

## 5. 라벨 (3-dim)

```
label_claude / label_llama / label_gpt
```

각 LLM의 해당 herb top-20에 candidate가 있으면 1, 없으면 0.

**판단이 아니라 기록이다.** "생강이 대추와 어울리는가"를 사람이 정하는 것이 아니라,
JSON에 있는지 없는지만 본다. 따라서 라벨링 비용이 0이다.

### 분포

| | positive | 비율 |
|---|---:|---:|
| Claude | 251 | 6.79% |
| Llama | 244 | 6.60% |
| GPT | 264 | 7.14% |
| 합집합 | 458 | 12.39% |

세 축 부재 1,451쌍 제외 후 학습 대상 2,245쌍, positive rate 약 17.6%.

### negative의 해석 한계

`label = 0`은 세 가지를 구분하지 못한다.

```
(a) LLM이 안 어울린다고 판단했다
(b) 순위가 밀려 top-20에 못 들었다
(c) LLM이 그 재료를 떠올리지 않았다
```

top-K 데이터의 본질적 한계(implicit feedback)이며, 리포트에 명시해야 한다.

> 본 연구의 negative는 "LLM이 top-20에 포함하지 않음"을 의미하며,
> 해당 조합이 실제로 부적합함을 뜻하지 않는다.

## 6. 알려진 데이터 품질 이슈

검증 과정에서 발견한 원본 오류들. 리포트의 재현성 항목에 사용 가능.

| 대상 | 오류 | 판정 근거 | 처리 |
|---|---|---|---|
| TCM-MKG D7 | 姜黄(강황)을 `Cold`로 기재 | 통설 溫. 같은 속 莪术은 정상 Warm. 郁金(寒)과 혼동 | 원본 유지 + sensitivity 분석 |
| TCM-MKG D6 | 회향 학명 *Pimpinella anisum* | 茴香은 *Foeniculum vulgare*가 통설 | 기록 |
| TCM-MKG D6 | 当归 영문명 "Japanese Angelica Root" | 중국종인데 일본종 명칭 | 기록 |
| 자동 매핑 | 아몬드 → 桃仁(복숭아씨) | 학명 오류 의심. 방제 264회로 오염 영향 큼 | **기각** |
| 자동 매핑 | 양고기 → 羊脬(방광) | 부위 불일치 | **기각** |
| 자동 매핑 | 멜론 → 甜瓜蒂(꼭지) | 최토약, 과육과 정반대 | **기각** |
| 자동 매핑 | 메이플시럽 → 五龙皮(수피) | 부위·가공 불일치 | **기각** |
| 자동 매핑 | 생선 → 鱼鳔(부레) | 부위 불일치 | **기각** |
| 자동 매핑 | 파슬리 → 欧芹菜籽(씨앗) | 잎/씨앗 불일치 | **기각** |
| 브리지 매핑 | water→水芹, egg→凤凰衣, rice→稻草 | 부분 토큰 매칭 | 158→89개 정제 |
| RecipeNLG | `date` = 대추야자(*Phoenix dactylifera*) | 한방 대추는 *Ziziphus jujuba* | 오염 명시 |
| RecipeNLG | `arrowroot` ≠ `kudzu` | *Maranta* vs *Pueraria*, 무관한 식물 | 분리 |
| FlavorGraph | `kitchenette_embeddings.pkl` | README 미언급, 별개 모델(KitchenNette) | 정식 300D로 교체 |

**부위 모호성**

- **우엉**: 식용은 뿌리(牛蒡根), 본초는 종자(牛蒡子)만 사용. 뿌리는 방제 0회.
- **연꽃**: 13개 부위 약재로 분리(연자육·연자심·하엽·연근 등). 소스마다 다른
  부위를 상정한다 — Claude는 연근·연자육, Llama는 연잎·찹쌀.

**월계수**: 대한민국약전·본초학 미수재. 성미귀경 없음. `mask_prop`이 항상 0이며,
**마스크 메커니즘의 자연 대조군** 역할을 한다.

## 7. 축별 예측력 (참고)

학습 결과에서 확인된 각 축 단독 성능. LOHO 14-fold × seed 5, PR-AUC.

| 축 | feature 수 | PR-AUC | 무작위(0.135) 대비 |
|---|---:|---:|---:|
| 조리 | 3 | 0.273 | 2.02× |
| 향미 | 2 | 0.213 | 1.58× |
| 전통 | 4 | 0.189 | 1.40× |
| 전체 3축 | 9 | 0.362 | 2.68× |

**전통 축이 feature 개수는 가장 많지만 예측력은 가장 낮다.**
차원 수가 아니라 라벨과의 상관이 성능을 결정한다.

> 이 값들은 **전체 test set 기준**이다. 각 축의 활성 쌍 수가 다르므로
> (조리 38.3% / 향미 51.5% / 전통 28.5%) 커버리지를 통제한 재계산이 필요하다.
> 전통 축은 71.5%의 쌍에서 입력이 전부 0인 채로 채점된 값이다.
